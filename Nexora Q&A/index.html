Data Security — Basics
06
How is user data stored and where does it live?Security
▾

All user data is stored in Supabase, which runs on PostgreSQL hosted on AWS infrastructure. Supabase provides enterprise-grade security out of the box.

Data storage breakdown:

  • User profiles, settings, preferences → Supabase PostgreSQL (encrypted at rest)
  • Brand deals, scheduled posts, earnings → Supabase PostgreSQL with Row Level Security
  • Media uploads, avatars, images → Supabase Storage (S3-compatible, private URLs)
  • OAuth access tokens → Supabase PostgreSQL, encrypted with AES-256 before storing
  • Passwords → Never stored in plain text. Handled entirely by Supabase Auth using bcrypt hashing
All data is encrypted at rest using AES-256. All data in transit uses TLS 1.3 (HTTPS). No data is ever stored unencrypted.
07
How do you ensure each user can only see their own data?Security
▾

We use a three-layer security model to ensure per-user data isolation:

1
Frontend Session Check
ProtectedRoute component checks Supabase session before rendering any page. No session → redirect to /login immediately.
2
Query-Level Scoping
Every database query includes .eq('user_id', uid) where uid comes from the authenticated session. No query ever fetches data without this filter.
3
Supabase Row Level Security (RLS)
Database-level enforcement using auth.uid() = user_id policies. Even if someone bypasses the frontend completely, the database rejects any request for data that doesn't belong to the authenticated user.

RLS is the most critical layer — it runs inside PostgreSQL itself, meaning it is impossible to bypass regardless of what the frontend or API sends. It is the same technology used by large SaaS companies like Notion, Linear, and Vercel.

08
How are passwords stored? Can your team read user passwords?Security
▾

No — it is technically impossible for anyone on our team to read user passwords.

Here is exactly how it works:

  • When a user sets a password, it is immediately hashed using bcrypt with a unique per-user salt before any storage occurs.
  • The hash is a one-way cryptographic function — it cannot be reversed to reveal the original password.
  • When logging in, the password the user types is hashed again and compared to the stored hash. If they match, access is granted — but the original password is never stored or transmitted.
  • This process is handled entirely by Supabase Auth, which is built on the same cryptography used by major banks and enterprises.
This means that even if our entire database were leaked, passwords would be useless to attackers. The only way to crack bcrypt hashes is brute force, which takes millions of years per password with modern hardware.

How are social media OAuth tokens (like YouTube, Instagram credentials) stored safely?Security
▾

OAuth tokens are the keys that allow Nexora OS to post on a user's behalf. Protecting them is critical.

  • We never ask for social media passwords. OAuth means users authorize us through the platform's official screen (e.g., Google's login page), and the platform gives us a limited-access token.
  • Tokens are encrypted with AES-256 on our backend before being stored in the database.
  • Tokens are stored with user_id and are protected by Row Level Security — one user's tokens cannot be accessed by another user's account.
  • Tokens have limited scope — we only request the minimum permissions needed (e.g., "post content" and "read analytics" — never "access contacts" or "read emails").
  • When a user disconnects a platform, the token is immediately and permanently deleted from our database.
  • Users can also revoke access directly from each platform's security settings (Google Account Permissions, etc.) as an additional safety layer.

Advanced Security Architecture
11
What is your complete security architecture stack?Security
▾

Our security architecture operates at seven levels:

  • Transport Security: TLS 1.3 on all connections (HTTPS everywhere). HSTS headers enforce HTTPS and prevent protocol downgrades.
  • Authentication: Supabase Auth with JWT tokens, bcrypt password hashing, and Google OAuth 2.0. Tokens expire after inactivity and are invalidated on logout.
  • Authorization: Row Level Security in PostgreSQL ensures database-level access control. Every table has RLS policies tied to auth.uid().
  • API Security: Supabase anon key is public by design (safe because RLS handles authorization). Service role key is server-side only, never exposed to the client.
  • Secret Management: All API keys (Groq, Stripe, Resend) stored as environment variables. Never committed to Git. GitHub secret scanning enabled.
  • Encryption: AES-256 for data at rest. TLS 1.3 for data in transit. bcrypt for passwords. AES-256 for OAuth token storage.
  • Deployment Security: Netlify HTTPS, automatic security headers, DDoS protection at the CDN level.

How do you protect against SQL Injection attacks?Attack Defense
▾

SQL injection is one of the most common attacks on web applications. It works when an attacker puts malicious SQL code into a form field (like a login box) hoping the server will execute it and dump the database.

Nexora OS is protected at multiple levels:

  • Supabase Client (Primary Defense): The Supabase JavaScript client uses parameterized queries by default. This means user input is always treated as data, never as executable SQL code. It is structurally impossible for user input to become a SQL command.
  • PostgREST Layer: Supabase's PostgREST API layer parses and validates all queries before they reach PostgreSQL, adding another layer of sanitization.
  • Row Level Security: Even if an injection somehow bypassed the above two layers (theoretically impossible), RLS would still prevent the attacker from accessing any data outside their own user_id scope.
Example: If an attacker types ' OR '1'='1 into a username field, the Supabase client treats it as a literal string search for that username, not as SQL. The database never sees it as a command.

How do you protect against Cross-Site Scripting (XSS) attacks?Attack Defense
▾

XSS attacks work by injecting malicious JavaScript into a web page. If a user can get their code to run in another user's browser, they can steal session tokens, read private data, or impersonate users.

Nexora OS defenses:

  • React's JSX escaping: React automatically escapes all dynamic content rendered in JSX. If a malicious user's name contains <script>alert('hacked')</script>, React renders it as literal text, not executable JavaScript.
  • No use of dangerouslySetInnerHTML: We avoid React's dangerouslySetInnerHTML which would bypass escaping. All content is rendered through safe JSX.
  • Content Security Policy (CSP) headers: CSP headers tell the browser to only execute scripts from trusted sources (our domain and approved CDNs). Even if an attacker injects a script tag, the browser refuses to run it.
  • HttpOnly cookies: Session tokens are stored in HttpOnly cookies (where applicable) which JavaScript cannot access, preventing token theft via XSS.

How do you handle DDoS (Distributed Denial of Service) attacks?Attack Defense
▾

A DDoS attack floods a server with millions of fake requests to crash it, making the service unavailable for real users. It's one of the most common attacks on public-facing web services.

Nexora OS has multiple layers of DDoS protection:

  • Netlify CDN (Frontend): The Nexora OS web app is served from Netlify's global CDN network with 200+ edge locations worldwide. DDoS traffic hitting the frontend is absorbed by the CDN's massive distributed infrastructure — not a single origin server.
  • Supabase Rate Limiting: Supabase automatically rate-limits database API requests per IP address and per user. Excessive requests are blocked before they reach the database.
  • Netlify Bot Protection: Netlify's infrastructure filters known malicious bots and attack patterns at the CDN edge before requests even reach our application.
  • API Rate Limiting (FastAPI Backend): Our Python backend implements rate limiting — if an IP sends too many requests in a short time, subsequent requests are rejected with a 429 error.
  • Cloudflare (planned): For additional protection, we are evaluating adding Cloudflare as a reverse proxy, which provides enterprise-grade DDoS mitigation including automatic attack detection and mitigation.
15
How do you protect against brute force login attacks?Attack Defense
▾

A brute force attack is when an attacker tries thousands of password combinations per second until they find the right one — like a robot trying every key on a keychain.

Nexora OS protections:

  • Supabase Auth Rate Limiting: Supabase automatically rate-limits login attempts. After a certain number of failed attempts from the same IP, further attempts are temporarily blocked.
  • bcrypt Slow Hashing: bcrypt is intentionally slow to compute — it takes ~100ms per password check. For a legitimate user, this is imperceptible. For an attacker trying a million passwords, 100ms × 1,000,000 = over 27 hours. This makes brute force attacks computationally infeasible.
  • Google OAuth (Recommended path): Users who sign in with Google bypass password-based attacks entirely — Google handles their authentication with 2FA, device verification, and anomaly detection.
  • Account Lockout: Repeated failed login attempts trigger account lockout notifications sent to the registered email, alerting the real user of suspicious activity.
  • Strong Password Requirements: Minimum 8 characters enforced at the API level, not just the frontend.
16
How do you protect against CSRF (Cross-Site Request Forgery) attacks?Attack Defense
▾

CSRF attacks trick a logged-in user's browser into making unauthorized requests — for example, a malicious website silently posting on the user's Instagram while they're browsing.

Nexora OS protections:

  • JWT Bearer Tokens: All API requests require a JWT token in the Authorization header. Browser-based cross-site requests cannot automatically include custom headers, so CSRF attacks cannot include the JWT and are rejected.
  • SameSite Cookie Attribute: Session cookies are set with SameSite=Strict, meaning they are never sent with cross-site requests.
  • CORS Policy: Our FastAPI backend only accepts requests from allowed origins (nexoraos.online and localhost during development). Requests from unknown domains are rejected entirely.
  • Supabase's built-in CSRF protection: The Supabase client library handles CSRF token generation and validation automatically.

What happens if Nexora OS itself gets hacked and there is a data breach?Attack Defense
▾

We take this scenario seriously and have a documented incident response plan:

Immediate response (0–2 hours):

  • Isolate the affected systems to stop the breach from spreading
  • Revoke all active session tokens, forcing all users to re-authenticate
  • Rotate all API keys and secrets immediately
  • Engage Supabase's security team if the database is involved

Within 72 hours:

  • Notify all affected users via email explaining what happened, what data was accessed, and what steps to take
  • Publish a public incident report on our status page
  • Report to relevant data protection authorities if required by law

Why the damage is limited even in a worst case:

  • Passwords cannot be cracked from our database (bcrypt)
  • Payment details are never stored (Stripe holds them)
  • OAuth tokens are AES-256 encrypted — useless without our encryption key
  • Social media passwords are never stored — only limited-scope tokens
Worst case breach exposure: Email addresses, display names, niche preferences, and follower counts. No passwords, no payment info, no social media credentials.

How do you handle database backups and disaster recovery?Data
▾

Data loss is treated as seriously as a security breach. Our backup and recovery strategy:

  • Automated Daily Backups: Supabase performs automated daily backups of the entire database. Backups are stored in encrypted form in geographically separate AWS regions.
  • Point-in-Time Recovery: On Supabase's Pro plan, we can restore the database to any point in time within the last 7 days — down to the second. If we accidentally delete a table at 3:42 PM, we can restore it to 3:41 PM.
  • Backup Testing: Backups are tested monthly by restoring to a staging environment to verify they work correctly. An untested backup is not a real backup.
  • Multi-region redundancy: Supabase's underlying AWS infrastructure provides redundancy across multiple availability zones — a hardware failure in one data center does not affect availability.
  • Recovery Time Objective (RTO): In a worst-case disaster scenario, we can restore full service within 2–4 hours using our backup snapshots.
